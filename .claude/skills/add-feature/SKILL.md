# 新機能の追加 (完全自動実行モード)

**重要:** このワークフローは、ユーザーの介入なしに、開始から完了まで完全に自動で実行されるように設計されています。各ステップは完了後、ただちに次のステップへ移行してください。思考の途中でユーザーに確認を求めたり、作業を中断したりしないでください。

> **Walking Skeleton にも使う:** このフローは通常の機能追加だけでなく、`/setup-project` が最初の「動く土台（Walking Skeleton）」を実装する際にも再利用される。骨格の場合、実装対象は「端から端まで動く最小の縦切り1本」であり、あわせて CI・フィットネス関数を緑にする。手順は同一なので本コマンドを流用してよい。

> **前提（権限モード）:** 本ワークフローは実装ファイルの編集に加え、サブエージェント起動（`Task`）や品質チェックコマンド（`Bash`: test / lint / typecheck 等）を実行する。これらは `settings.json` の `allow` では事前承認しきれない（コマンドの実体がプロジェクト依存のため）。「無停止」で完走させたい場合は、**acceptEdits もしくはバイパス系の権限モードで起動する**こと。通常モードでは各コマンドで承認プロンプトが入り、その都度停止する。

> **並列実行（worktree モード / 任意）:** 複数機能を同時に進める場合、機能ごとに git worktree を切り、それぞれ別セッションの Claude Code で本コマンドを実行する（手順とレシピは `README.md`「並列開発と夜間バッチ運用」）。
> - **並列可能かは `docs/backlog.md` の「依存」列で判定する**: 依存先が `[x] 完了` の未着手機能だけを並列に着手する（`bash scripts/progress.sh` が「並列着手可能」を列挙し、依存違反を警告する）
> - `.steering/` は gitignore されているため worktree ごとに独立する。backlog の状態更新（`scripts/backlog-state.sh`）は各ブランチ内で行い、マージで統合する
> - 同じファイルを触る機能同士は並列にしない（依存列に依存を書くのが先）。判断に迷ったら直列で実行する

**実装対象の機能:** $ARGUMENTS

> `$ARGUMENTS` が空の場合は、実装したい機能名をユーザーに一度だけ確認してから開始する。

---

## ステップ1: 準備とコンテキスト設定

1. 現在のタスクコンテキストを確立する:
  - 機能名: `$ARGUMENTS`（上記「実装対象の機能」で与えられた値）
  - 日付: `[現在の日付をYYYYMMDD形式で取得]`
  - ステアリングディレクトリパス: `.steering/[日付]-[機能名]/`
2. 上記ステアリングディレクトリを作成する。
3. 以下の3つの空ファイルを作成する:
  - `[ステアリングディレクトリパス]/requirements.md`
  - `[ステアリングディレクトリパス]/design.md`
  - `[ステアリングディレクトリパス]/tasklist.md`
4. `docs/backlog.md` が存在する場合、着手を記録する（全体俯瞰の鮮度維持）:
  - `bash scripts/backlog-state.sh start "[機能名]" "[日付]-[機能名]"` を実行する（状態列の正規更新経路。Edit での直接変更は hooks がブロックする）
  - 該当行が無ければこのコマンドが行を追加する（バックログ外の着手＝スコープ追加なので、ステップ8の完了報告で明示する）
  - backlog が無い場合（`/setup-project` を経ていないプロジェクト）はスキップしてよい

## ステップ2: プロジェクト理解

1. `CLAUDE.md`を読み、プロジェクトの全体像を把握する。
2. `docs/`ディレクトリ内の永続ドキュメントを確認し、関連する設計思想やアーキテクチャを理解する。

## ステップ3: 既存パターンの調査 (サブエージェント起動)

**調査は `pattern-scout` サブエージェントに委譲し、本体のコンテキストには圧縮された調査ブリーフだけを取り込む**（調査で読む大量のコードが本体コンテキストを埋めると、後半タスクの生成品質が劣化するため）。

1. `Task`ツールを使用し、`pattern-scout`サブエージェントを起動する。
  - `subagent_type`: "pattern-scout"
  - `description`: "Existing pattern scouting"
  - `prompt`: "機能 `[機能名]` の実装計画に向けて、**お前自身の定義（調査の規律・報告形式）に従って**既存パターンを調査し、調査ブリーフを返してください。調査規律（ソースルートの解決・依存性ルールでの選別・踏襲禁止の区別）はエージェント定義に一元化されているので、ここでは再定義しない。"
2. 返ってきた調査ブリーフをステップ4（計画）の入力とする。ブリーフで「踏襲禁止」とされたパターンは真似ず、正しい依存方向で実装する。

## ステップ4: 計画フェーズ (ステアリングファイルの自動生成)

1. `Skill('steering')`を**計画モード**で実行し、ステップ1で作成した3つのファイル (`requirements.md`, `design.md`, `tasklist.md`) の内容を生成する。
  - **タスク粒度は「1タスク = 1つの振る舞い（= 1つのテストで駆動できる単位）」にする。** これはステップ5の Red-Green-Refactor サイクルを回すための前提。「〇〇クラスを実装」のようなコンポーネント単位ではなく、「〇〇が××のとき△△を返す」という振る舞い単位に割る。粒度の正本は tasklist テンプレートと steering モード2「テスト駆動規律」に従う。
  - **機能スコープの上限: tasklist の振る舞いはおおむね5〜15個に収める。** 1回の無停止実行に積む量が多いほどコンテキストが埋まり、後半のタスクほど生成品質が劣化する（タスク粒度をいくら細かくしても、この劣化は防げない）。15個を超える場合、この機能は1回の実行には大きすぎるサイン。**端から端まで動く縦切りを保ったまま機能を分割し、今回のステアリングは最初の縦切りに限定する。** 分割したら `bash scripts/backlog-state.sh split "[機能名]" "[今回の縦切り名]" "[残りの縦切り名]..."` を実行し、全体俯瞰（`docs/backlog.md`）に反映する（該当行を今回の縦切り名に改名し、残りを `[ ] 未着手` で行追加。残りがチャットログにしか残らないと、セッション終了とともに俯瞰から消えるため）。残りの縦切りは tasklist に入れず、ステップ8の完了報告で「次回の /add-feature 候補」として提示する（行の追加は意図の領域なので、スコープ変更として明示しユーザーの確認機会を作る）。
2. **このステップが正常に完了したら、決して停止せず、ただちにステップ5に進むこと。**

## ステップ5: 実装ループ (steering 実装モードへ完全委譲 / テスト駆動)

**`tasklist.md`の全タスクが `[x]` になるまで、各タスクを Red-Green-Refactor で実装するループ。ループ手順・進捗記録・スキップ規律・テスト駆動規律の正本は `steering` スキルのモード2に一元化されている。ここでは委譲のみを行い、規律を再定義しない。**

1. `Skill('steering')`を**実装モード**で実行し、`tasklist.md`の全タスクを消化する。
  - ループ手順（次タスク特定 → `[ ]`→`[-]`着手記録 → **🔴Red→🟢Green→🔵Refactor** で実装 → `[-]`→`[x]`完了記録 → 次へ）は steering モード2に従う。
  - **テスト駆動規律**（失敗するテストを先に書く／通す最小限だけ実装する／各タスクごとにテストを緑にしてから次へ）は、steering モード2の「テスト駆動規律（正本）」に従う。**テストはこのループの中で振る舞いごとに走る。ステップ7まで先送りしない。**
  - **タスク完遂規律**（未完了タスクを残さない・スキップは技術的理由のみ・大きすぎるタスクは分割・ユーザーに判断を仰がない）は、steering モード2の「タスク完遂規律（正本）」に従う。
  - **分割トリガー（測定可能なサイン）**: Red を3回試してもテストが緑にならない、または1タスクの変更が3ファイルを超えたら、そのタスクは大きすぎるサインとみなす。正本「タスクが大きすぎる場合」に従い、振る舞い単位のサブタスクへ分割してから続行する。
2. 実装中は常に `Skill('development-guidelines')` のコーディング規約を遵守する。
3. **全タスクが `[x]`（または正当な技術的理由でスキップ）になり、各タスクのテストが緑になったら、決して停止せず、ただちにステップ6に進むこと。**

## ステップ6: 実装検証 (サブエージェント起動)

1. `tasklist.md`の全タスクが完了したことを最終確認する。
2. `Task`ツールを使用し、`implementation-validator`サブエージェントを起動して品質を検証する。
  - `subagent_type`: "implementation-validator"
  - `description`: "Implementation quality validation"
  - `prompt`: "今回実装した `[機能名]` に関する全ての変更点の品質を、**お前自身の定義（検証観点）に従って**検証してください。対象ファイルは `[実装したファイルのパスリスト]` です。検証観点・優先順位・既存違反の扱いは validator 定義に一元化されているので、ここでは再定義しない。"

**このステップが正常に完了したら、決して停止せず、ただちにステップ7に進むこと。**

## ステップ7: 全体リグレッションと静的ゲート（最終セーフティネット）

> ここは「初めてテストを走らせる場所」ではない。ステップ5で振る舞いごとにテストは緑になっている。
> このステップは、**スイート全体の回帰（他機能を壊していないか）と、機械的な品質ゲートの最終確認**を担う。

1. プロジェクトの品質チェックコマンドを「テスト（スイート全体）→ Lint → 型チェック → アーキテクチャ境界検査（依存方向）」の順に実行し、全てパスすることを確認する。
  - **各コマンドの実体（言語・ツール名）はここに書かない。** `CLAUDE.md` の「技術スタック」または `docs/development-guidelines.md` を参照して解決する（このコマンドは順序と種類のみを規定する）。
  - 重複検出やPRサイズなど追加の機械ゲートが `development-guidelines.md` に定義されていれば、あわせて実行する。
2. いずれかのコマンドでエラーが発生した場合は、問題を分析し、修正する。**テストが失敗した場合は、まず失敗を再現するテストを足してから修正する（Red→Green を守る）。** 修正後、再度このステップを実行する。

**このステップが正常に完了したら、決して停止せず、ただちにステップ8に進むこと。**

## ステップ8: 振り返りとドキュメント更新

1. `Skill('steering')`を**振り返りモード**で実行し、`tasklist.md`に申し送り事項を記載する。
  - 実装完了日
  - 計画と実績の差分
  - 学んだこと
  - 次回への改善提案

2. **描写（description）ドキュメントを、今回の実装ぶんだけ育てる（必須）。** `.claude/rules/document-management.md`「意図と描写の分離（正本）」に従い、描写はコードから導出して鮮度を保つ。今回のコードを真実の源として、以下を追記する:
   - `docs/functional-design.md`: 実装した機能の設計を追記（スタブから肉付けする）
   - `docs/glossary.md`: 今回登場した新しいドメイン用語を追記
   - `docs/repository-structure.md`: ディレクトリ構成が変わった場合、現実に合わせて更新
   - `docs/backlog.md`: `bash scripts/backlog-state.sh done "[機能名]"` で該当行を `[x] 完了` に更新（存在する場合。tasklist.md の全タスク `[x]` が完了の根拠。乖離は `scripts/progress.sh` が機械検出する）。**ステップ4で機能を分割した場合、行は改名済みなので `[今回の縦切り名]` を指定する。** また完了報告に、分割で追加した残りの縦切りの行を「次回の /add-feature 候補（スコープ変更）」として明示する

2.5. **描写ドリフト監査（機械検査 → サブエージェント起動）。** 2段構えで検証する:
   - **まず機械検査**: `bash scripts/check-drift.sh` を実行する（記載パスの実在・用語の出現の決定論的検査）。❌ が出たら**ドキュメント側を修正**して緑にしてから次へ進む
   - **次に意味の監査**: `Task`ツールで`drift-auditor`サブエージェントを起動し、機械では判定できない意味的な乖離がないことを検証する。
   - `subagent_type`: "drift-auditor"
   - `description`: "Description drift audit"
   - `prompt`: "今回の実装で更新した描写ドキュメントを、**お前自身の定義（監査観点・判定の非対称性）に従って**実コードと突き合わせて監査してください。今回実装したファイルは `[実装したファイルのパスリスト]` です。"
   - **不合格（[必須]乖離あり）の場合、修正するのはドキュメント側**（真実の源はコード）。修正後に再監査し、合格するまで繰り返す。

3. 今回の変更がプロジェクトの「意図」（PRD／`architecture.md` の芯）に影響を与えるか判断する。**影響がある場合でも、意図ドキュメントを直接編集しない。** 意図はユーザーが舵を握る領域であり（`.claude/rules/spec-driven-workflow.md`「承認ゲートの方針（正本）」）、実装が意図を書き戻すのは依存の逆流になる。代わりに:
   - 変更案（どのドキュメントの・どの記述を・なぜ・どう変えるべきか）を tasklist.md の振り返りに記録する
   - 完了報告の末尾に「**意図ドキュメントへの変更提案**」として提示し、ユーザーの承認を仰ぐ（本フローはここで完了するため、無停止の原則には反しない）

## 完了条件

このワークフローは、以下の全ての条件を満たした時点で自動的に完了となる。
- ステップ5: `tasklist.md`の全てのタスクが完了状態（`[x]`または正当な理由でスキップ）になっており、各タスクが Red-Green-Refactor で実装され、対応するテストが緑になっている（テスト省略は正本の例外に該当し注記済みの場合のみ）。
- ステップ6: `implementation-validator`サブエージェントの検証をパスする。
- ステップ7: `test`（スイート全体）, `lint`, `typecheck`, アーキテクチャ境界検査（依存方向）の全てのコマンドがエラーなく成功する。
- ステップ8: `tasklist.md`に申し送り事項が記載されており、`scripts/check-drift.sh` が緑かつ`drift-auditor`サブエージェントの描写ドリフト監査に合格している（意図への影響がある場合は、変更提案が振り返りに記録され、完了報告に「意図ドキュメントへの変更提案」として含まれている）。

この完了条件を満たすまで、自律的に思考し、問題解決を行い、作業を継続すること。
