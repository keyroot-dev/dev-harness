# Steering スキル

ステアリングファイル（`.steering/`）に基づいた実装を支援し、tasklist.mdの進捗管理を確実に行うスキルです。

## スキルの目的

- ステアリングファイル（requirements.md, design.md, tasklist.md）の作成支援
- tasklist.mdに基づいた段階的な実装管理
- **進捗の自動追跡とtasklist.md更新の強制**
- 実装完了後の振り返り記録

## 使用タイミング

このスキルは以下のタイミングで使用してください：

1. **作業計画時**: ステアリングファイルを作成する時
2. **実装時**: tasklist.mdに従って実装する時
3. **検証時**: 実装完了後の振り返りを記録する時

## モード1: ステアリングファイル作成

### 目的
新しい機能や変更のためのステアリングファイルを作成します。

### 手順

1. **ステアリングディレクトリの確認**
   ```
   現在の日付を取得し、`.steering/[YYYYMMDD]-[機能名]/` の形式でディレクトリを作成
   ```

2. **永続ドキュメントの確認**
   - `docs/product-requirements.md`
   - `docs/functional-design.md`
   - `docs/architecture.md`
   - `docs/repository-structure.md`
   - `docs/development-guidelines.md`

   これらを読んで、プロジェクトの方針を理解する

3. **テンプレートからファイル作成**

   以下のテンプレートを読み込み、プレースホルダーを具体的な内容に置き換えてファイルを作成:

   - `.claude/skills/steering/templates/requirements.md` → `.steering/[日付]-[機能名]/requirements.md`
   - `.claude/skills/steering/templates/design.md` → `.steering/[日付]-[機能名]/design.md`
   - `.claude/skills/steering/templates/tasklist.md` → `.steering/[日付]-[機能名]/tasklist.md`

4. **tasklist.mdの詳細化**

   requirements.mdとdesign.mdに基づいて、tasklist.mdを詳細化:
   - 各フェーズのタスクを具体的に記述
   - サブタスクも明確に
   - 実装の順序を明記
   - 振る舞いは PRD の受け入れ条件（EARS 記法。正本: `prd-writing/SKILL.md`「受け入れ条件（EARS 記法）」）から導出する。EARS の1条件が tasklist の1振る舞い（= 1テスト）に対応する

## モード2: 実装（最重要）

### 目的
tasklist.mdに従って実装を進め、**進捗を確実にドキュメントに記録**します。

### 🚨 タスク完遂規律（このスキルの正本 / Single Source of Truth）

> このリポジトリにおける「タスクをどう完遂するか・いつスキップを許すか」の規律は、**ここ一箇所だけで定義する**。
> `add-feature` コマンドや本スキルの他セクション（ステップ4.5 等）は、この規律を**再定義せず参照する**。
> 規律を見直すときは、必ずこの節だけを変更すればよい状態に保つこと。

**タスクの状態は3値で記録する（tasklist.md が常に真実を語るための状態機械）**:

```
[ ] 未着手 ──着手時──▶ [-] 実装中 ──テストが緑になった時のみ──▶ [x] 完了
```

`[x]` は「完了の証拠（テスト緑）」がある時にだけ付ける。**着手時に `[x]` を付けてはならない**
── セッションが中断した場合、未完了の作業が「完了」として記録に残り、記録が嘘をつく。

**MUST（必須）**:
- tasklist.md を常に開いた状態で実装する
- タスク着手時（`[ ]`→`[-]`）・完了時（`[-]`→`[x]`）に必ず Edit ツールでリアルタイムに更新する（複数タスクをまとめて更新しない）
- **tasklist.md の全タスクが `[x]` になるまで作業を継続する**
- 進捗の正は tasklist.md。TodoWrite は揮発的な補助メモにすぎない

**NEVER（禁止）**:
- tasklist.md を見ずに実装を進める／更新せずに次のタスクへ進む
- 「時間の都合」「難しい」「後で」「別タスクとして実施予定」を理由にタスクをスキップする
- 未完了タスク（`[ ]`）を残したまま振り返りを書く・作業を終了する

**タスクが大きすぎる場合**: スキップせず、小さなサブタスクに分割して tasklist.md に追加し、1つずつ完了させる。

**スキップが許されるのは「技術的理由」のみ**:
- 実装方針の変更により、機能自体が不要になった
- アーキテクチャ変更により、別の実装方法に置き換わった
- 依存関係の変更により、タスクが実行不可能になった
- 上位の設計変更により、このタスクが無意味になった

  スキップ手順:
  1. 該当行を `- [x] ~~タスク名~~（理由: 技術的理由を具体的に記述）` の形式で更新する
  2. 振り返りセクションに、なぜ不要になり何に置き換わったかを記録する

**NG例 / 正しい完了の形**:
```markdown
## 実装後の振り返り
**実装しなかったタスク**:
- テストの実装（時間の都合により別タスクとして実施予定） ❌ 絶対にダメ
```
正しい完了 = 全タスクが `[x]` ／「実装しなかったタスク」の記述が無い ／ 方針変更があればその理由が明記されている。

### 🔴🟢🔵 テスト駆動規律（このスキルの正本 / Single Source of Truth）

> このリポジトリにおける「コードとテストをどの順で書くか」の規律は、**ここ一箇所だけで定義する**。
> `add-feature` コマンドや tasklist テンプレートは、この規律を**再定義せず参照する**。
>
> **なぜテストを先に書くのか（AI時代の根拠）**: 生成コードは「その場しのぎ」に流れやすく、書き直し（churn）と重複が増えることが実測されている。テストを先に定義することで、(1) 各振る舞いに対する速いフィードバックループが生まれ、(2) テスタブルな設計が強制され、(3) 「動いた感」ではなく「動く証拠」で進捗を測れる。テストは実装の**検品**ではなく**駆動力**である。

**TDDの三原則（MUST）**:

1. 失敗するテストを書くまで、プロダクトコードを書いてはならない。
2. 失敗させるのに十分なだけのテストしか書いてはならない（コンパイル不能も失敗に含む）。
3. 現在失敗しているテストを通すのに十分なだけのプロダクトコードしか書いてはならない。

**Red → Green → Refactor サイクル（各タスク = 1つの振る舞いで必ず回す）**:

```
  ┌──────────────────────────────────────────────────────────┐
  │ 🔴 Red    : その振る舞いを表す失敗するテストを1つ書く      │
  │             → テストを実行し「意図した理由で失敗」を確認    │
  │ 🟢 Green  : 通すための最小限の実装だけを書く              │
  │             → テストを実行し緑になることを確認            │
  │ 🔵 Refactor: 重複を除き整える（テストは緑のまま維持）      │
  │             → 再実行して緑を確認                          │
  └──────────────────────────────────────────────────────────┘
```

**NEVER（禁止）**:
- テストを書かずにプロダクトコードを書き始める
- 複数の振る舞いを一気に実装してから、後でまとめてテストを書く（テスト・ラスト）
- Red（失敗の確認）を飛ばして、いきなり実装とテストを同時に書く
- Refactor で重複を放置したまま次のタスクへ進む

**テストを書かない例外（技術的理由のみ / 振り返りに記録）**:
- 自明な設定・宣言のみで振る舞いを持たない（例: 定数定義、DIの結線のみ）
- フレームワーク境界の薄いアダプタで、統合テスト側でカバーされる
- （UI/インフラ詳細など）ユニットテストが不適で、別レイヤーのテストで担保される

  この場合も、なぜテストを書かなかったかを tasklist.md の該当行に注記する。

### 実装フロー

#### ステップ1: tasklist.mdを読み込む

```
Read('.steering/[日付]-[機能名]/tasklist.md')
```

全体のタスク構造を把握し、次に着手すべきタスクを特定する。

#### ステップ2: TodoWriteでタスク管理開始

tasklist.mdの内容に基づいてTodoWriteツールでタスクリストを作成:
- これはClaude Code内部の補助的なメモ
- **tasklist.mdこそが正式なドキュメント**

#### ステップ3: タスクループ（各タスクで繰り返す）

**3-1. 次のタスクを確認**
```
tasklist.mdを読み、次の未完了タスク（`[ ]`）を特定
（中断からの復帰時は、実装中のまま残っている`[-]`のタスクを最優先で再開する）
```

**3-2. タスク開始をtasklist.mdに記録（必須）**
```
Editツールを使って、tasklist.mdの該当行を`[ ]`→`[-]`（実装中）に更新

例:
old_string: "- [ ] タイトルが空のとき、タスク作成を拒否しValidationErrorを返す"
new_string: "- [-] タイトルが空のとき、タスク作成を拒否しValidationErrorを返す"
```

**注意**: タスク名は「振る舞い」単位で書かれているはずだ（正本: tasklist テンプレート）。
「〇〇クラスを実装」「〇〇Serviceを作る」のようなコンポーネント単位の名前を見つけたら、
着手前に振る舞い単位のサブタスクへ分割する。

**重要**: Editツールを実行した直後に、更新が成功したことを確認する。

**3-3. TodoWriteでもステータス更新**
```
TodoWriteツールで該当タスクを"in_progress"に変更
```

**3-4. 実装を実行（Red → Green → Refactor で回す）**

「テスト駆動規律（正本）」に従い、このタスクの振る舞いを次のマイクロサイクルで実装する。
開発ガイドライン（docs/development-guidelines.md）のコーディング規約とテストの書き方（Given-When-Then）を遵守する。

```
🔴 Red     : この振る舞いを表す失敗するテストを1つ書く
             → テストコマンドを実行し、「意図した理由で」失敗することを確認
🟢 Green   : テストを通す最小限の実装だけを書く
             → テストコマンドを実行し、緑になることを確認
🔵 Refactor: 重複除去・命名整理・依存方向の是正を行う（テストは緑のまま）
             → 再実行して緑を確認
```

- 1タスクに複数の振る舞いが含まれる場合は、振る舞いごとにこのサイクルを繰り返す
  （＝タスクが大きすぎる兆候。正本「タスクが大きすぎる場合」に従いサブタスクへ分割する）。
- テストコマンドの実体（ツール名）はここに書かない。`CLAUDE.md`「技術スタック」または
  `docs/development-guidelines.md` を真実の源として解決する。

**3-5. タスク完了をtasklist.mdに記録（必須）**
```
Red→Green→Refactor が緑で完了したことを確認してから、
必ずEditツールで該当行を`[-]`→`[x]`に更新して完了を記録
（テストが緑になる前に`[x]`を付けない）

サブタスクがある場合はサブタスクも個別に更新する
```

> テストを書かなかった場合（正本「テストを書かない例外」に該当する場合のみ）は、
> 該当行に理由を注記する: `- [x] {タスク名}（テスト省略: {技術的理由}）`

**3-6. TodoWriteでもステータス更新**
```
TodoWriteツールで該当タスクを"completed"に変更
```

**3-7. 次のタスクへ**
```
ステップ3-1に戻る
```

#### ステップ4: フェーズ完了時の確認

各フェーズ（例: フェーズ1、フェーズ2）が完了したら:

1. **tasklist.mdを読み込んで進捗確認**
   ```
   Read('.steering/[日付]-[機能名]/tasklist.md')
   ```

2. **完了したタスクを確認**
   - すべてのタスクが`[x]`になっているか
   - 見落としたタスクがないか

3. **ユーザーに報告**
   ```
   「フェーズ1が完了しました。tasklist.mdの進捗を確認してください。」
   ```

   > **報告のみで停止しない。** 無停止フロー（/add-feature 等）から呼ばれている場合、
   > ユーザーの応答を待たず、ただちに次のフェーズへ進む。

#### ステップ4.5: 全タスク完了チェック（必須）

**全フェーズの実装完了後、振り返りを書く前に必ず実行**:

1. **tasklist.mdを読み込む**
   ```
   Read('.steering/[日付]-[機能名]/tasklist.md')
   ```

2. **未完了タスク（`[ ]`）・実装中タスク（`[-]`）が1つも残っていないか確認**

3. **未完了タスクが見つかった場合**は、本スキルの「🚨 タスク完遂規律（正本）」に従って処理する:
   - 実装できるなら → ステップ3（タスクループ）に戻って実装する
   - 大きすぎるなら → サブタスクに分割し、1つずつ完了させる
   - 技術的理由で不要になったなら → 正本のスキップ手順で処理する

   （「時間の都合により別タスクとして実施予定」と振り返りに書くことは禁止。詳細・例はすべて正本を参照）

4. **全タスクが`[x]`になっていることを確認してからステップ5へ進む**

#### ステップ5: 全タスク完了後

1. **最終確認**
   ```
   Read('.steering/[日付]-[機能名]/tasklist.md')
   ```

   すべてのタスクが`[x]`になっていることを確認

2. **振り返りセクションに記録**
   ```
   Editツールでtasklist.mdの「実装後の振り返り」セクションを更新:
   - 実装完了日
   - 計画と実績の差分
   - 学んだこと
   - 次回への改善提案
   ```

### 実装中のセルフチェック

5タスクごとに以下を確認:

- [ ] tasklist.mdを最近更新したか？（最後の更新から5タスク以内）
- [ ] 進捗がドキュメントに反映されているか？（Read toolで確認）
- [ ] ユーザーがtasklist.mdを見て進捗が分かるか？

## モード3: 振り返り

### 目的
実装完了後、tasklist.mdに振り返りを記録します。

### 手順

1. **tasklist.mdを読み込む**
   ```
   Read('.steering/[日付]-[機能名]/tasklist.md')
   ```

2. **振り返り内容を作成**
   - 実装完了日
   - 計画と実績の差分（計画と異なった点）
   - 学んだこと（技術的な学び、プロセス上の改善点）
   - 次回への改善提案

3. **Editツールで更新**
   ```
   tasklist.mdの「実装後の振り返り」セクションを更新
   ```

4. **ユーザーに報告**
   ```
   「振り返りをtasklist.mdに記録しました。内容を確認してください。」
   ```

## トラブルシューティング

### tasklist.mdの更新を忘れた場合

もし実装中にtasklist.mdの更新を忘れていることに気づいたら:

1. **即座に更新を実行**
   ```
   Read('.steering/[日付]-[機能名]/tasklist.md')
   完了したタスクを特定し、すべてEditツールで`[x]`に更新
   ```

2. **ユーザーに報告**
   ```
   「tasklist.mdの更新が遅れていたため、現在の進捗を反映しました。」
   ```

3. **再発防止**
   - 次のタスクから確実に更新する
   - 5タスクごとのセルフチェックを徹底

### tasklist.mdと実装の乖離

計画と実装が大きく異なる場合:

1. **tasklist.mdに注釈を追加**
   ```
   Editツールで該当タスクに注釈を追加:
   「- [x] タスク名（実装方法を変更: 理由）」
   ```

2. **必要に応じて新しいタスクを追加**
   ```
   Editツールで新しいタスクを追加
   ```

3. **design.mdも更新**
   ```
   設計の変更が大きい場合はdesign.mdも更新
   ```

## チェックリスト（最重要）

実装前に必ず確認:

- [ ] tasklist.mdを読み込んだか？
- [ ] 次のタスクを特定したか？
- [ ] タスク開始時にEditツールで更新したか？

実装後に必ず確認:

- [ ] タスク完了時にEditツールで更新したか？
- [ ] tasklist.mdの進捗を確認したか？
- [ ] ユーザーが見て進捗が分かる状態か？

## スキルの効果

このスキルを正しく使用すると:

- ✅ tasklist.mdが常に最新の進捗を反映
- ✅ ユーザーが進捗を一目で把握できる
- ✅ ドキュメントと実装の乖離がなくなる
- ✅ 振り返りが容易になり、次回の改善につながる
- ✅ プロジェクト履歴として価値がある記録が残る

## 重要なリマインダー

🚨 **このスキルの最も重要な役割は、tasklist.mdの進捗管理を確実に行うことです。**

- TodoWriteは揮発的なメモ（ユーザーには見えない）
- **tasklist.mdこそが永続的なドキュメント（ユーザーが見る）**

実装中は常に「ユーザーがtasklist.mdを見たときに進捗が分かるか？」を自問してください。
