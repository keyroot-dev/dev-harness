# 進捗報告（progress-report）

進捗の真実（`docs/backlog.md` と `.steering/*/tasklist.md`）から、PM向けの報告を**その都度導出**するスキルです。

```
docs/backlog.md（機能単位の全体俯瞰）          ← ズームアウト
        │ steering列のリンク
        ▼
.steering/*/tasklist.md（振る舞い単位の真実）  ← ズームイン
        │ scripts/progress.sh が横断集計＋乖離検査
        ▼
本スキル ──▶ PM向け報告をチャットに生成（保存しない・都度導出）
```

## 原則（このスキルの正本）

- **報告は導出ビュー。手で盛らない**: tasklist.md に `[x]` として記録されていない完了を報告に書いてはならない。逆に、記録済みの事実を省略しない。
- **報告ファイルを保存しない**: `docs/` は頻繁に更新されない北極星のみを置く場所（`CLAUDE.md`「ドキュメント管理の原則（正本）」）。時点スナップショットはチャットに出すだけでよい。過去の完了履歴は git が持っており、`scripts/progress.sh --since` でいつでも再導出できる。
- **嘘の上に報告を組み立てない**: 鮮度検査が ❌ を出したら、まず乖離（backlog／tasklist の更新漏れ）を修正してから報告を作る。

## 手順

### ステップ1: 機械集計（真実の取得）

1. 報告期間の起点を決める:
   - `$ARGUMENTS` に日付（YYYY-MM-DD）があればそれを使う
   - 省略時は7日前（macOS: `date -v-7d +%F` / GNU: `date -d '7 days ago' +%F`）
2. 集計スクリプトを実行する:
   ```bash
   bash scripts/progress.sh --since [起点日付]
   ```
   出力: 機能単位の状態集計・作業単位のタスク集計・鮮度検査・期間内に完了した振る舞い。

### ステップ2: 乖離があれば先に修正する

鮮度検査が ❌（非ゼロ終了）の場合、報告を作る前に乖離を修正する:

- **真実の優先順位は tasklist.md**（振る舞いレベルの一次記録）。backlog の状態列をそれに合わせて修正する
- tasklist 側の記録漏れ（実装済みなのに `[ ]` のまま等）が疑われる場合は、steering スキルの「トラブルシューティング」に従って実態を反映する
- 修正後、`bash scripts/progress.sh` を再実行し ✅ を確認してからステップ3へ進む

### ステップ3: 文脈の取得

1. `docs/backlog.md` を読む（全体の現在地・次にやる未着手機能の優先度）
   - 無い場合: 全体俯瞰は不可。`.steering/` のみで報告し、報告末尾で backlog の種まき（`templates/backlog.md` 参照）を提案する
2. 実装中（`[-]` を含む）の steering の `tasklist.md` を読み、残タスクと、振り返り・スキップ注記からブロッカー・リスクを拾う

### ステップ4: 報告の生成（チャットに出力）

以下のフォーマットで報告をチャットに出力する。数字と完了項目はすべてステップ1の機械集計から転記する（自分で数え直さない・創作しない）。

```markdown
# 進捗報告（YYYY-MM-DD / 対象期間: MM-DD 〜 MM-DD）

## サマリ

（3行以内。何が進み、いま何をしていて、注意すべき点は何か）

## 全体の現在地

機能: 完了 n ／ 実装中 m ／ 未着手 k（全 N 機能）
タスク: [██████████░░░░░░░░░░] 50%（x / y 完了）

## この期間に完了した振る舞い（テスト緑 = 動く証拠つき）

- ✅ （tasklist.md の振る舞い文をそのまま使う）

## 進行中

- [-] （実装中の振る舞い）… 残り j タスク

## 次にやること（backlog の優先度順）

- （未着手機能を優先度順に）

## ブロッカー・リスク

- （振り返り・スキップ注記から。無ければ「なし」）
```

**書き方の注意**:
- 読者は PM。tasklist の振る舞い文（「〇〇が××のとき△△する」）は非エンジニアにも読める形で書かれているので、そのまま使うのが最も正確で速い
- 進捗バーは20文字（`█` = 完了割合、`░` = 残り）で描く
- 「完了」の根拠は常にテスト緑（動くソフトウェア）。「たぶん動く」「ほぼ完了」という中間表現を使わない — 状態は3値しかない
