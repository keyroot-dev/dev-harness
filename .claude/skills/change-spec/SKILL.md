# 仕様変更（開発途中の意図の改訂）

開発が始まった**後**に「作るもの」を変えたくなったときのワークフローです。**意図の変更は人が舵を握り、決まった変更の波及は機械的に自走する**。

> **設計の正本はここで再定義しない。** いつ承認で止まるかは `.claude/rules/spec-driven-workflow.md`「承認ゲートの方針（正本）」、意図と描写の扱いは `.claude/rules/document-management.md`「意図と描写の分離（正本）」に従う。本スキルはそれを「仕様変更」という事象に適用する手順にすぎない。

**変更内容:** $ARGUMENTS

> `$ARGUMENTS` が空、または「何を・なぜ変えたいか」が読み取れない場合は、その2点だけを一度確認してから開始する。

## 全体の流れ

```
 変更要求「やっぱり〇〇にしたい」
     │
     ▼
 ① 現在地の取得 ──▶ ② 影響分析 ──▶ ③ 意図の改訂案 ──▶ 🛑 承認ゲート ──▶ ④ 伝播 ──▶ ⑤ ハンドオフ
   (progress.sh)     (3バケット)    (PRD/アーキ芯)     (意図の改訂)     (自走)     (/add-feature)
```

なぜこの順か: 意図（何を・なぜ）の変更は方向転換であり、承認なしに走り出すと、間違った方向がコード・バックログ・進行中の計画へ一気に波及する。逆に、承認された意図を各記録に反映する伝播は機械的作業なので、止まる理由がない。

## 触ってよいもの・いけないもの

```
 意図（intent）           PRD・architecture.md の芯・backlog の行と優先度
   → 本スキルの改訂対象。ただし承認ゲートを通してから確定する

 進行中の計画             .steering/*/（実装中 [-] のもの）
   → 承認後に修正する（steering の規律に従う）

 描写（description）      functional-design.md・glossary.md・repository-structure.md・
                          development-guidelines.md・backlog の状態列
   → ❌ 本スキルでは触らない
```

**描写を触らない理由**: コードがまだ旧仕様のままである限り、旧仕様を描写しているドキュメントこそが「真実」である。仕様変更を先取りして描写を書き換えると、実在しないコードの記述＝「詳細な嘘」を自ら作ることになる（context poisoning）。描写はコードが変わるとき（`/add-feature` のステップ8）に育ち、drift-auditor が鮮度を守る。

## ステップ1: 現在地の取得（機械集計）

1. `bash scripts/progress.sh` を実行し、機能単位・タスク単位の現在地と乖離の有無を得る
2. `docs/backlog.md` を読む（機能×状態×steering の地図）
3. 実装中（`[-]`）の steering があれば、その `requirements.md`・`design.md`・`tasklist.md` を読む
4. 変更対象に関係する範囲で `docs/product-requirements.md`・`docs/architecture.md` を読む

> 鮮度検査が ❌ の場合、嘘の上に影響分析を組み立てない。先に乖離を修正してから進む（真実の優先順位は tasklist.md）。

## ステップ2: 影響分析

変更が触れる**意図のセクション**（PRDの機能要件・受け入れ条件・非機能要件・スコープ外／アーキ芯の依存の向き・レイヤ境界・技術スタック）を特定した上で、影響を受ける機能を backlog の状態で3バケットに分類する:

```
 backlog の状態    仕様変更が意味すること                伝播先
──────────────────────────────────────────────────────────────────────
 [ ] 未着手    →  意図の書き換えだけで済む（最安）    →  PRD・backlog の行
 [-] 実装中    →  進行中の計画の修正が要る            →  steering 3ファイル
 [x] 完了      →  動いているコードの手直しが要る      →  backlog にリワーク行
──────────────────────────────────────────────────────────────────────
```

- **アーキ芯（技術スタック・レイヤ境界）に触れる変更**は、原理的に全バケットへ波及しうる。影響レポートでその旨を明示し、必要なリワークを機能単位に割って洗い出す
- **意図に触れない変更**（リファクタリング・実装詳細の変更）だと判明したら、本スキルの適用を打ち切り、通常の作業（会話 or `/add-feature`）として扱う — 仕様変更の儀式を不要な場面に持ち込まない

## ステップ3: 意図の改訂案の作成

1. 対象ドキュメントに応じてスキルをロードして改訂する（既存の構造と内容を維持しながら更新）:
   - PRD → `Skill('prd-writing')`
   - アーキ芯 → `Skill('architecture-design')`
2. **廃止する機能は記述を消さず「スコープ外」へ移し、理由を一行添える**。将来の読者（人間とLLM）が「なぜ無いのか」を読めるようにするため。詳細な経緯は git 履歴が持つ
3. **ユーザーに見せる前に自己レビューを1周かける**: `Task`ツールで`doc-reviewer`サブエージェントを起動し（`subagent_type`: "doc-reviewer"）、改訂したドキュメントをレビューさせる。[必須]指摘を自己修正してから提示する（承認の往復を減らすための工程であり、ゲートを増やすものではない）

## ステップ4: 承認ゲート（意図の改訂）

**改訂した意図は必ずユーザー承認を待つ**（正本: `.claude/rules/spec-driven-workflow.md`「承認ゲートの方針（正本）」）。以下を1つの報告にまとめて提示する:

```markdown
## 仕様変更の改訂案

### 変更の要約
（何が・なぜ変わるか。3行以内）

### 意図の差分
- `docs/product-requirements.md`: （変更したセクションと変更前→後の要点）
- `docs/architecture.md`: （芯に触れた場合のみ）

### 影響範囲
- 未着手: （意図の書き換えのみで済む機能）
- 実装中: （計画修正が要る steering と、無効化/追加されるタスクの要点）
- 完了済み: （リワークが要る機能と作業の要点）

### 承認後に行う伝播
（backlog の行の増減・優先度変更、steering の計画修正の一覧）
```

提示後は次のように伝えて**承認まで待機する**:

```
「仕様変更の改訂案ができました。意図の差分と影響範囲を確認してください。
承認いただけたら、バックログと進行中の作業計画へ変更を伝播します。」
```

- 修正指示があれば改訂案に反映し、再度このゲートで待つ
- **却下されたら意図ドキュメントを変更前の状態に戻して終了する**（git があるので巻き戻しは安全にできる）

## ステップ5: 承認後の伝播（自走）

承認された意図を各記録へ反映する。ここからは判断を含まない機械的作業なので、途中でユーザーに確認を求めない。

1. **backlog（意図の部分）を更新する**:
   - 新機能・リワークは行を追加する（状態は `[ ] 未着手`、steering 列は `-`）。リワーク行は元機能と区別できる名前にする（例: 「期限入力の必須化（仕様変更: 20260713）」）
   - 不要になった**未着手**の行は削除してよい（行の存在は意図であり、意図が消えた）
   - **完了済み（`[x]`）の行は消さない** — 過去に完了した事実の記録であり、消すと progress.sh の俯瞰と git 履歴が語る事実が食い違う
   - 優先度の変更を反映する
   - **状態列は触らない**（機械更新の領分。正本: `.claude/rules/document-management.md`「意図と描写の分離（正本）」のハイブリッド節）
2. **実装中（`[-]`）の steering の計画を修正する**:
   - `requirements.md`・`design.md` に変更を反映する
   - `tasklist.md` は steering の規律（正本: `.claude/skills/steering/SKILL.md` モード2「タスク完遂規律（正本）」）に従って修正する:
     - 無効化された振る舞いは `- [x] ~~タスク名~~（理由: 仕様変更により〇〇に置き換え）` の形式でスキップ処理する（「上位の設計変更」は正本が認める技術的理由）
     - 新しい振る舞いをタスクとして追加する（1タスク = 1つの振る舞い）
     - **完了済み（`[x]`）タスクの記録は書き換えない** — テスト緑で完了した事実は仕様変更後も事実
3. **完了済み機能のコードは触らない**: リワークは backlog に行として積むだけ。実装は `/add-feature` が TDD で行う
4. **検証**: `bash scripts/progress.sh` を実行し、鮮度検査 ✅（乖離なし）を確認する

> 本スキル自身は steering ディレクトリを作らない。実装を伴わないドキュメント作業だからであり、実装が要る変更はすべて backlog の行に落ち、`/add-feature` が steering を作る。

## ステップ6: 完了報告とハンドオフ

```markdown
# 仕様変更の伝播が完了しました

## 更新したもの
- 意図: （PRD・アーキ芯の変更セクション）
- backlog: （追加/削除/優先度変更した行）
- steering: （修正した計画とタスクの増減）

## 次のアクション
- 実装中だった作業の再開: （修正済み tasklist に従って続行できる旨と対象 steering）
- リワーク・新機能の実装: /add-feature [最優先の行の機能名]
```

## 完了条件

- 改訂した意図ドキュメントがユーザーに承認されている（却下時は巻き戻して終了）
- backlog の行と優先度が承認内容どおりに更新され、状態列には触れていない
- 実装中の steering がある場合、tasklist が steering の規律どおりに修正されている（無効化はスキップ形式・完了記録は不変）
- 描写ドキュメント（functional-design.md 等）に変更を先取りした記述が**無い**
- `bash scripts/progress.sh` が ✅（乖離なし）で終了する
