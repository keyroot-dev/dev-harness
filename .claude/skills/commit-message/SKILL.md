# コミットメッセージ生成（commit-message）

変更内容（diff）から規約準拠のコミットメッセージを導出し、ブランチ規律を守ってコミットするスキルです。

```
ブランチ確認 ────────▶ 規約と実例の取得 ──▶ 変更の把握 ──▶ メッセージ生成 ──▶ コミット
（main/devなら起点を     （正本＋git log）    （diffが真実）   （型式に当てはめ）  （log -1で確認）
  ユーザーに確認して切る）
```

## 原則（このスキルの正本）

- **規約はこのスキルに書き写さない**: コミット規約の正本は、プロジェクトの `docs/development-guidelines.md` の「コミットメッセージ規約」。まだ生成されていなければ、テンプレート既定の `development-guidelines/guides/process.md`「コミットメッセージの規約」（Conventional Commits）に従う。
- **main へ直接コミットしない**: main への変更は PR を経由する（正本: `development-guidelines/guides/process.md`「ブランチ戦略」）。main / master / dev などの統合ブランチ上にいたら、コミットの前に必ずトピックブランチを切る。どこから切るかは人の判断なので、ユーザーに確認する。
- **メッセージは diff の描写**: diff に無いことを書かない・diff にある本質を落とさない。読者は `git log --oneline` を流し読みする未来の開発者である。
- **サマリ1行を基本とする**: 1行で「何を・なぜ」が伝わるなら本文（body）は書かない。本文は破壊的変更（BREAKING CHANGE）や、diff から読み取れない理由の説明が必要なときだけ最小限に書く。

## 手順

### ステップ1: ブランチの確認（main なら切る）

```bash
git branch --show-current
```

- **main / master / dev などの統合ブランチ上の場合**: 直接コミットせず、トピックブランチを切ってから進む。ただし**起点をどこにするかはユーザーに確認する**（例:「main と dev のどちらから切りますか?」。AskUserQuestion があれば使う）。起点が決まったら `git switch -c <type>/<短い内容> <起点ブランチ>` で作成する（未コミットの変更はそのまま持っていける）。名前の例: `feat/task-priority`、`fix/login-error`、`docs/update-readme`。
  - 確認を省略してよいのは、統合ブランチが1本しか存在しない場合（`git branch -a` で main しか無い等）。その場合はそこから切る旨を報告して進む。
- **既にトピックブランチ上の場合**: そのまま進む。ただしブランチ名と今回の変更の関心事が明らかに別物なら、新しいブランチを提案する。

### ステップ2: 規約と実例の取得

1. `docs/development-guidelines.md` が存在すれば「コミットメッセージ規約」セクションを読む（プロジェクト固有の規約が最優先）
2. 無ければ `development-guidelines/guides/process.md`「コミットメッセージの規約」を読む（type 一覧・フォーマットの正本）
3. `git log --oneline -10` で実際のスタイルを確認し、それに合わせる（scope の粒度・言語・サマリの長さ。実例は規約の生きた解釈である）

### ステップ3: 変更の把握（diff が真実）

```bash
git status
git diff --staged   # ステージ済みが無ければ git diff
```

- ステージ済みの変更が無い場合: 変更内容を確認し、今回の関心事に属するファイルをステージする
- **1コミット = 1関心事**: 無関係な変更が混ざっている場合は、コミットを分割する（メッセージに「〜と〜」「あわせて〜も」が必要になったら混在のサイン）

### ステップ4: メッセージの生成

`<type>(<scope>): <subject>` 形式に当てはめる:

- **type**: 正本の一覧から diff の性質で選ぶ。迷ったら「この変更が無かったら何が困るか」で判定する（機能が無い→feat、壊れたまま→fix、読めない→docs…）
- **scope**: 変更の主対象を1語で（省略可。git log の実例に粒度を合わせる）
- **subject**: 日本語で、変更の意図が1行で伝わる長さ（50字前後を目安）。「〜を追加」「〜を修正」「〜に変更」のように結果が分かる述語で終える

### ステップ5: コミットと確認

```bash
git commit -m "<生成したメッセージ>"
git log -1 --oneline
```

コミット後、ブランチ名とメッセージをユーザーに報告する（main に居なかったこと・どのブランチに積んだかが分かるように）。

## 例

**例1: 新しいスキルファイルを追加した diff**
```
feat(commit-message): コミット規約に従いメッセージを生成するスキルを追加
```

**例2: 既存スクリプトの検査漏れを直した diff**
```
fix(check-config): CRLF ファイルでフロントマター検査が素通りする問題を修正
```

**例3: 破壊的変更を含む diff（本文が必要な例外ケース）**
```
feat(task): Task型にpriority必須フィールドを追加

BREAKING CHANGE: 既存のタスクデータはマイグレーションが必要
```
