# 技術調査（tech-research）

実装中の「調べたい」を、**最新・公式の一次情報**から裏取りして整理するスキルです。

**なぜ記憶で即答しないか**: モデルの記憶にあるライブラリ知識は「古い可能性のある描写」であり、確信を持って間違える（古いAPIの幻覚）。このリポジトリの原則「描写は真実でなければ毒」（正本: `.claude/rules/document-management.md`「意図と描写の分離」）は調査にもそのまま当てはまる — 真実の源は現物のドキュメントであり、記憶は当たりを付けるためだけに使う。

## ステップ1: 調査質問を特定する

1. `$ARGUMENTS` から「何を知りたいか」「何を判断したいか」を1〜2文の質問に絞る。曖昧なら実装文脈（直前の会話・編集中のファイル）から補う
2. **プロジェクトの実バージョンを確認する**: `CLAUDE.md` の技術スタックと、依存マニフェスト（`package.json` / `go.mod` / `pyproject.toml` 等）を読む。「最新の情報」ではなく「**このプロジェクトが使うバージョンに一致した情報**」が必要 — v14 のプロジェクトに v15 の書き方を答えたら、それは詳細な嘘になる

## ステップ2: 情報源へルーティングする

調べたいことの種類で一次情報への経路を選ぶ:

```
 調べたいこと
   │
   ├─ ライブラリ/フレームワークの API・使い方・設定
   │    └─▶ Context7（resolve-library-id → query-docs）
   │        バージョン一致の公式docsを直接取得。Web検索より先にこれ
   │
   ├─ リリース情報・変更履歴・非推奨化・既知バグ
   │    └─▶ WebFetch で公式リポジトリの Release Notes / CHANGELOG / Migration Guide
   │
   ├─ エラーメッセージ・トラブルシュート
   │    └─▶ WebSearch で当たりを付け、公式 issue / docs を WebFetch で裏取り
   │        （ブログ・Q&Aサイトは手がかり。結論の根拠は一次情報に置く）
   │
   └─ 技術選定・ライブラリ比較
        └─▶ WebSearch で複数の候補と論点を集め、
            各候補の公式docs・リポジトリ（更新頻度・メンテ状況）で数値と事実を裏取り
```

**裏取りの規律**:
- 結論の根拠は必ず一次情報（公式ドキュメント・リリースノート・公式リポジトリ）に置く。二次情報（ブログ・Q&A）だけを根拠に断定しない
- 情報には**日付とバージョン**を確認する。日付のない情報・バージョン不明の情報は「鮮度不明」として扱う
- 2〜3回の検索・取得で確証が得られないときは、深追いせず「ここまで分かった／ここが未確認」を正直に区別して報告する

## ステップ3: 整理して出力する

調査結果はチャットに以下の形式で出力する。読者は「実装の手を止めて調べに来た人」— 最初の3行で判断できるようにする。

```markdown
# 調査: [質問]

## TL;DR

（結論を1〜3行。「〜すべき」「〜が原因」まで言い切る。言い切れないなら何が未確認かを書く）

## 図解

（概念・仕組み・データフローが関わる場合は ASCII 図で構造を示す。単純な事実確認なら省略可）

## 詳細

（根拠となる事実。各項目に出典を付ける）
- 事実A（出典: [公式docs URL]・vX.Y 時点・確認日 YYYY-MM-DD）

## このプロジェクトへの適用

（プロジェクトの実バージョン・既存構成を踏まえた具体的な適用方法。汎用論で終わらせない）

## 未確認・注意点

（裏取りできなかったこと、バージョン差で変わりうること。無ければ「なし」）
```

**書き方の注意**:
- **事実と推測を混ぜない**。一次情報で確認できたことだけを「詳細」に書き、推測は「未確認・注意点」へ
- 出典 URL と確認日を省略しない。後日この調査を読み返したとき、鮮度を判定できることが価値になる

## 調査結果の保存

- 基本は**チャットに出力するだけ**（都度導出。実装の判断材料であって北極星ではない）
- 技術選定の根拠など後で参照する価値がある場合のみ、ユーザーの求めに応じて `docs/ideas/` に保存する（役割の正本: `.claude/rules/document-management.md`「下書き・アイデア」）。`docs/` 直下の正式ドキュメントには書かない
- 調査の結果、意図（PRD・アーキ芯）を変えたくなったら、このスキルで書き換えず `/change-spec` に引き継ぐ
