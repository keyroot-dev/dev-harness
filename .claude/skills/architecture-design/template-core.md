# アーキテクチャの芯 (Architecture Core)

> **これは「芯」= 意図(intent)であり、Walking Skeleton を1本通すのに必要な最小限の決定だけを書く。**
> - タイミング: `/setup-project` のPRD承認直後（コードの前）。
> - 詳細な構造・データフロー・部品カタログ・パフォーマンス/セキュリティ/スケーラビリティの具体は**本体（描写）**であり、骨格・機能実装の**後**に `template.md` に沿って育てる。ここに先取りで書かない（context poisoning を避ける）。
> - 依存性のルールの正本は `guide.md`「依存性のルール」。ここの図は生成物 `docs/architecture.md` を自己完結させるための出力であり、規則の二重定義ではない。

## 1. 何のシステムか（Screaming Architecture）

このシステムを**フレームワークではなくユースケースで**1〜2文で言い切る。

```
NG: "Node.js + TypeScript のプロジェクト"        ← フレームワークが主役
OK: "タスク管理システム (add / list / complete)"  ← ユースケースが主役
```

> [ここに、このシステムが何をするかを1〜2文で。技術名を主語にしない。]

## 2. コアのドメインとユースケース（MVP縦切りのぶんだけ）

Walking Skeleton で通す**最小の縦切り**に登場するものだけを書く。網羅しない。

### エンティティ（心臓部）

| エンティティ | 責務 | 守るべき不変条件 |
|-------------|------|------------------|
| [例: Task] | [タスクを表す] | [タイトルは1〜200文字 / 完了済みは再完了不可] |

### ユースケース（動詞で命名）

| ユースケース | 入力 | 出力 | 概要 |
|-------------|------|------|------|
| [例: CreateTask] | [title] | [Task] | [新規タスクを作成する] |

> 骨格に含めない機能は書かない。以後 `/add-feature` が本体（`architecture.md` / `functional-design.md`）に追記して育てる。

## 3. 依存性のルールとレイヤ境界（最重要 / 変えにくい決定）

**ソースコードの依存は必ず内側へ向ける。** 外側（詳細）が内側に依存し、内側は外側を一切知らない。

```
┌────────────────────────────────────────────┐
│ Frameworks & Drivers (詳細 / 最も外側)        │  DB / CLI / Web / FS
│  ┌──────────────────────────────────────┐  │
│  │ Interface Adapters                    │  │  Controller / Presenter / Repository実装
│  │  ┌────────────────────────────────┐  │  │
│  │  │ Use Cases (Interactor)         │  │  │
│  │  │  «interface» Repository ◄───────┼──┼──┼─ 境界I/FはUse Case側が所有
│  │  │  ┌──────────────────────────┐  │  │  │
│  │  │  │ Entities (心臓部)         │  │  │  │  ドメインの不変条件
│  │  │  └──────────────────────────┘  │  │  │
│  │  └────────────────────────────────┘  │  │
│  └──────────────────────────────────────┘  │
└────────────────────────────────────────────┘
        依存はすべて内向き ───→ 中心へ
```

- 具象（Repository実装等）を `new` してよいのは構成ルート（`main` / `index`）だけ。内側の層は依存を注入される。
- 境界を越える入出力はDTO（単純なデータ構造）で行い、Entity を外へ漏らさない。
- レイヤ構成（このプロジェクトで採用する層）: [例: entities / use-cases / adapters / infrastructure]

## 4. 技術スタック（骨格を立てるのに要る最小限）

Walking Skeleton をビルド・テストするのに必要な決定だけ。選定理由は簡潔に。

| 区分 | 選定 | 理由（簡潔に） |
|------|------|----------------|
| 言語 / ランタイム | [例: TypeScript 5.x / Node.js 24 LTS] | [理由] |
| パッケージマネージャー | [例: npm] | [理由] |
| テスト | [例: vitest] | [理由] |
| Lint / フォーマット | [例: eslint + prettier] | [理由] |
| 型チェック | [例: tsc --noEmit]（型のある言語のみ） | [理由] |
| アーキ境界検査（Fitness Function） | [例: dependency-cruiser] | 依存方向を機械強制する |

> ここは `CLAUDE.md`「技術スタック」節と一致させる（テスト・Lintコマンドの真実の源）。

## 5. 境界の機械強制（Fitness Function）の方針

依存性のルールは**ドキュメントの禁止で終わらせず、Walking Skeleton とあわせてCIで強制する**（違反でビルドを落とす）。

- 検査対象: `entities/` の外向き依存禁止 / `use-cases/` の詳細依存禁止 / 循環依存(ADP)禁止
- 具体的なルール設定は骨格作成時に用意し、`development-guidelines.md`「アーキテクチャ境界の強制」に描写する。
