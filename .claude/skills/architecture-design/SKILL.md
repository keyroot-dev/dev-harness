# アーキテクチャ設計スキル

このスキルは、高品質なアーキテクチャ設計書を作成するための詳細ガイドです。

## アーキテクチャは「芯」と「本体」の2段で育てる

`CLAUDE.md`「ドキュメント管理の原則（正本）」に従い、`architecture.md` は次の2段で作る。

- **芯（コア / 意図・intent）= コードの前に、薄く書く**
  - 内容: 依存の向き（依存はすべて内向き）・レイヤ境界・技術スタック（言語/ランタイム/テスト・Lintツール）
  - 目的: Walking Skeleton を1本通すのに必要な最小限の「決定」だけ。詳細な構造や部品カタログはここに書かない。
  - タイミング: `/setup-project` のPRD承認直後。
- **本体（描写・description）= Walking Skeleton と機能実装の後に、コードから育てる**
  - 内容: 実際に生まれたコンポーネント構成・データフロー・具体的な依存関係
  - タイミング: 骨格の後、および `/add-feature` が機能を追加するたびに追記。憶測で先取りしない。

> `/setup-project` から呼ばれた初回は「芯」だけを書く。以降の肉付けは骨格・機能実装の後に行う。

## 前提条件

- **芯を書くとき**: `docs/product-requirements.md` (PRD) があれば十分。
- **本体を育てるとき**: Walking Skeleton または実装済みコードを真実の源として読む。

> 注意: `functional-design.md` は本スキルの前提ではない（機能設計は描写ドキュメントで、アーキ芯より後に育つ）。芯は PRD の意図から直接引く。

## 既存ドキュメントの優先順位

**重要**: `docs/architecture.md` に既存のアーキテクチャ設計書がある場合、
以下の優先順位に従ってください:

1. **既存のアーキテクチャ設計書 (`docs/architecture.md`)** - 最優先
   - プロジェクト固有の技術選定と設計が記載されている
   - このスキルのガイドより優先する

2. **このスキルのガイド** - 参考資料
   - 汎用的なテンプレートと例
   - 既存設計書がない場合、または補足として使用

**新規作成時**: このスキルのテンプレートとガイドを参照
**更新時**: 既存設計書の構造と内容を維持しながら更新

## 出力先

作成したアーキテクチャ設計書は以下に保存してください:

```
docs/architecture.md
```

## テンプレートの参照

アーキテクチャ設計書は「芯」と「本体」で使うテンプレートが異なる。用途に応じて選ぶ:

- **芯（コア / 意図）を書くとき**（`/setup-project` のPRD承認直後、コードの前）:
  - テンプレート: ./template-core.md ← Walking Skeleton を1本通すのに必要な最小限だけ
- **本体（描写）を育てるとき**（骨格・機能実装の後）:
  - テンプレート: ./template.md ← 詳細な構成・データフロー・品質特性などを追記

いずれの場合も、依存性のルールなどの原則は次のガイドを参照する:
- ガイド: ./guide.md（依存性のルールの正本）
