## アーキテクチャパターン

### クリーンアーキテクチャ（依存はすべて内向き）

```
┌────────────────────────────────────────────┐
│ Frameworks & Drivers (詳細 / 最も外側)        │  DB / CLI / Web / FS
│  ┌──────────────────────────────────────┐  │
│  │ Interface Adapters                    │  │  Controller / Presenter
│  │  ┌────────────────────────────────┐  │  │  Repository実装
│  │  │ Use Cases (Interactor)         │  │  │
│  │  │  «interface» Repository         │  │  │  ← 境界I/FはここでDefine
│  │  │  ┌──────────────────────────┐  │  │  │
│  │  │  │ Entities (心臓部)         │  │  │  │  ドメインの不変条件
│  │  │  └──────────────────────────┘  │  │  │
│  │  └────────────────────────────────┘  │  │
│  └──────────────────────────────────────┘  │
└────────────────────────────────────────────┘
        依存はすべて内向き ───→ 中心へ
```

#### Entities（心臓部 / 最も内側）
- **責務**: 企業全体で通用する最高レベルのビジネスルールと不変条件
- **許可される依存**: なし（何にも依存しない）
- **禁止**: フレームワーク・DB・UIへの依存

#### Use Cases（Interactor）
- **責務**: アプリケーション固有のビジネスルール。境界インターフェース（Repository等）を**所有**する
- **許可される依存**: Entities、自身が定義した抽象インターフェース
- **禁止**: 具体的なDB・フレームワーク・UIへの依存

#### Interface Adapters
- **責務**: 外側の形式（HTTP / CLI / DBスキーマ）と内側の形式を相互変換する
- **許可される依存**: Use Cases、Entities
- **禁止**: ビジネスルールの実装
- **境界を越えるデータ**: Use Case との受け渡しは**DTO（単純なデータ構造）**で行う。
  Entity を境界の外へ素通ししない（内側のドメイン変更が外側を壊さないため）

#### Frameworks & Drivers（詳細 / 最も外側）
- **責務**: DB・ファイルI/O・Web等の詳細。Use Case層のインターフェースを**実装**する（DIP）
- **許可される依存**: 内側すべて（依存の矢印は内向き）
- **禁止**: 内側がここを知ること

> **依存性逆転（DIP）の要**: Repository等の境界インターフェースは Use Case 層が所有し、
> Frameworks & Drivers 層がそれを実装する。これによりDB・フレームワークは「あとで差し替え可能な詳細」になる。

### 境界の機械的強制（Fitness Function）

依存性のルールは**ドキュメントの記述だけでなくLint/CIで強制**します（違反でビルドを落とす）。

| 検査内容 | ツール（例） |
|---------|-------------|
| `entities/` の外向き依存禁止 / `use-cases/` の詳細依存禁止 / 循環依存禁止 | [例: dependency-cruiser / ArchUnit / import-linter] |

> 具体的なルール設定は `development-guidelines.md` の「アーキテクチャ境界の強制」に置く。

## コンポーネント構成

機能領域（コンポーネント）の分割と、その間の依存方向を定義します。

| コンポーネント | 含むもの | 依存先（→は内向き/安定方向のみ） | 安定度 |
|--------------|---------|------------------------------|--------|
| [例: task-management] | entities / use-cases / adapters / infrastructure | [shared] | [安定/不安定] |
| [名称] | [責務] | [依存先] | [安定度] |

> **原則**: 依存グラフは非循環（ADP）に保ち、依存は「より安定・より抽象」な方向へ向ける（SDP/SAP）。
> 機能領域をまたぐ依存は `shared/` 経由のみとし、コンポーネント間の循環は作らない。
