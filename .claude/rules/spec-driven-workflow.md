# スペック駆動開発ワークフロー

## 基本フロー

1. **ドキュメント作成**: 永続ドキュメント(`docs/`)で「何を作るか」を定義
2. **作業計画**: ステアリングファイル(`.steering/`)で「今回何をするか」を計画
3. **実装**: tasklist.mdに従って実装し、進捗を随時更新
4. **検証**: テストと動作確認
5. **更新**: 必要に応じてドキュメント更新

## 承認ゲートの方針（正本）

**「いつ人間の承認で止まり、いつ自走するか」はここで一元的に定義する。** 各コマンド・スキルはこの方針を再定義せず、これに従う。

- **セットアップ（`/setup-project`）**: 熟慮が必要なため、意図を固める点と、動く土台を確認する点に承認ゲートを置く。
  - **ゲート1（意図）**: 起点となる `product-requirements.md` は**作成後に必ずユーザー承認を待つ**。
  - PRD承認後、`architecture.md` の芯（依存の向き・レイヤ境界・技術スタック）と `docs/backlog.md`（機能単位の全体俯瞰）を生成する。どちらも意図（intent）なのでPRDを真実の源として連続生成してよい。
  - **ゲート2（動く土台）**: 続けて Walking Skeleton（動く縦切り1本＋CI・フィットネス関数が緑）を実装し、**動くことをユーザーに見せて一拍置く**。フィットネス関数には**アーキ境界検査（`templates/fitness/README.md` のキットから1本）を必ず含める**（ゼロ本はゲート2不合格）。
  - **描写（description）ドキュメント**（`repository-structure.md`・`functional-design.md`・`glossary.md`・`development-guidelines.md`）は、骨格が動いた**後**に、コードを真実の源として導出／種まきする。憶測でコードの前に書かない（理由は `.claude/rules/document-management.md` の「意図と描写の分離（正本）」を参照）。
- **実装（`/add-feature`）**: 計画から完了まで**自走する（途中でユーザー判断を仰がない）**。
  - 判断に迷う事態が起きても、`steering` の「タスク完遂規律（正本）」に従って自律的に対処する。
  - 実装後、育つ描写ドキュメント（機能設計・用語集等）を、その機能ぶんだけ追記して育てる。
- **仕様変更（`/change-spec`）**: 開発開始後に意図（PRD・アーキ芯・backlog の行と優先度）を変えたくなったとき。影響分析と改訂案の作成までは自走し、**改訂した意図は必ずユーザー承認を待つ（ゲート3: 意図の改訂）**。承認後の伝播（backlog・進行中 steering の計画修正）は機械的作業なので自走する。描写ドキュメントは変更を先取りして書き換えない（コードが変わるときに `/add-feature` が育てる）。

> 使い分けの理由: 意図（何を・なぜ）は方向性を決めるので人間が舵を握り、動く土台も一度確認する。描写（どこに何が）はコードから導出すべきもので、実装はその意図に従う「実行」なので規律に沿って一気通貫で進める。

## 承認ゲートでの伝え方

承認ゲート（上記「承認ゲートの方針（正本）」で定義した3点）に達したら、必ず立ち止まり、明確に伝える。ゲート以外（アーキ芯や、骨格後の描写ドキュメント生成）では止まらず連続で進めてよい。

**ゲート1（意図＝PRD）で待つとき**:
```
「product-requirements.md の作成が完了しました。内容（意図）を確認してください。
承認いただけたら、アーキテクチャの芯と Walking Skeleton に進みます。」
```

**ゲート2（動く土台＝Walking Skeleton）で待つとき**:
```
「Walking Skeleton が動く状態になりました（CI・フィットネス関数〔アーキ境界検査を含む〕も緑）。
実行して確認してください。承認いただけたら、骨格を元に残りの描写ドキュメントを生成します。」
```

**ゲート3（意図の改訂＝仕様変更）で待つとき**:
```
「仕様変更の改訂案ができました。意図の差分と影響範囲を確認してください。
承認いただけたら、バックログと進行中の作業計画へ変更を伝播します。」
```

## 実装前の確認

新しい実装を始める前に、必ず以下を確認:

1. CLAUDE.mdと `.claude/rules/` の規律を確認する（`paths` なしの規律は毎セッション自動ロード。`paths` 付きは該当ファイルを扱うときのみロードされるため、対象規律の有無を CLAUDE.md の索引表で確認する）
2. 関連する永続ドキュメント(`docs/`)を読む
3. Grepで既存の類似実装を検索
4. 既存パターンを理解してから実装開始

## ステアリングファイル管理

作業ごとに `.steering/[YYYYMMDD]-[タスク名]/` を作成:

- `requirements.md`: 今回の要求内容
- `design.md`: 実装アプローチ
- `tasklist.md`: 具体的なタスクリスト

命名規則: `20250115-add-user-profile` 形式

**作業計画・実装・検証時は`steering`スキルを使用してください。**

- **作業計画時**: `Skill('steering')`でモード1(ステアリングファイル作成)
- **実装時**: `Skill('steering')`でモード2(実装とtasklist.md更新管理)
- **検証時**: `Skill('steering')`でモード3(振り返り)

詳細な手順と更新管理のルールはsteeringスキル内に定義されています。

## 開発プロセス

### 初回セットアップ

1. このテンプレートを使用
2. `/setup-project` で PRD（意図）を作成・承認 → アーキ芯 → Walking Skeleton（動く土台）→ 骨格を元に描写ドキュメントを導出（承認ゲートの方針・意図と描写の分離の正本に従う）
3. `/add-feature [機能]` で機能実装

### 日常的な使い方

**基本は普通に会話で依頼してください:**

```bash
# ドキュメントの編集
> PRDに新機能を追加してください
> architecture.mdのパフォーマンス要件を見直して
> glossary.mdに新しいドメイン用語を追加

# 機能追加(定型フローはコマンド)
> /add-feature ユーザープロフィール編集

# 仕様変更(開発途中で作るものを変えたくなったら)
> /change-spec タスクの期限をオプションから必須に変更
> やっぱり認証はOAuthにしたい

# 進捗の確認・PM向け報告(全体俯瞰は docs/backlog.md を開くだけでも可)
> /progress-report
> 進捗どう?

# 詳細レビュー(詳細なレポートが必要なとき)
> /review-docs docs/product-requirements.md
```

**ポイント**: スペック駆動開発の詳細を意識する必要はありません。Claude Codeが適切なスキルを判断してロードします。
