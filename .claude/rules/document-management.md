# ドキュメント管理の原則

## 意図と描写の分離（正本 / Single Source of Truth）

**「どのドキュメントをコードの前に書き、どれをコードの後に書くか」はここで一元的に定義する。** 各スキル・コマンドはこの原則を再定義せず、これに従う。

ドキュメントには2種類ある。**この2種類を、書くタイミングで扱い分ける。**

- **意図（intent）ドキュメント = コードの「前」に書く**
  - 該当: `product-requirements.md`、`architecture.md` の芯（依存の向き・レイヤ境界・技術スタック）
  - 内容: 何を・なぜ作るか、制約、受け入れ条件、インターフェース、やらないこと
  - 理由: 意図を前もって詳しく与えると、LLMの生成品質が上がる（スペック駆動 / コンテキストエンジニアリング）。ここはケチらず詳しく書く。

- **描写（description）ドキュメント = コードの「後」に、コードから導出して書く**
  - 該当: `repository-structure.md`、`functional-design.md`、`glossary.md`、`development-guidelines.md`（骨格の設定から種を採る部分）
  - 内容: どこに何があるか、現在のモジュール構成、実際に使われているドメイン用語
  - 理由: 存在しないコードの描写を先に書くと「詳細な嘘」になり、それを真実として読んだLLMが確信を持って間違える（context poisoning）。**描写は真実でなければ武器にならない。** 真実の源はコードであり、描写はそこから導出して鮮度を保つ。

- **ハイブリッド: バックログ（`docs/backlog.md`）**
  - 行の存在と優先度は意図（PRD承認後に導出・人が舵を握る）。**状態列だけは描写**であり、`/add-feature` が着手・完了時に機械更新する（手で `[x]` にしない）
  - 進捗報告はここと `.steering/*/tasklist.md` から**都度導出**する（`/progress-report`）。報告書ファイルは保存しない — `docs/` は頻繁に更新されない北極星のみを置く場所であり、時点スナップショットは git 履歴からいつでも再導出できる

> 一言原則: **意図は分厚く前に、描写は真実として後から。** 真実を伴わない詳細は、規模を伴う毒（context poisoning）になる。

**描写の鮮度は機械で守る**: 描写ドキュメントと実コードの乖離（ドリフト）は、放置すればLLMへの毒になる。`scripts/check-config.sh` が参照リンク切れを機械検出するのと同じ思想で、描写と実装の乖離はフィットネス関数／CIで検出し、腐敗を防ぐ。進捗の乖離（backlog の状態列と tasklist の実態のズレ）は `scripts/progress.sh` が検出する。

## ディレクトリ構造と役割

### 永続的ドキュメント(`docs/`)

アプリケーション全体の「何を作るか」「どう作るか」を定義。基本設計を記述し、頻繁に更新されない、プロジェクト全体の「北極星」。

#### 下書き・アイデア（`docs/ideas/`）
- 壁打ち・ブレインストーミングの成果物
- 技術調査メモ
- 自由形式（構造化は最小限）
- `/setup-project`実行時に自動的に読み込まれる

#### 正式版ドキュメント
- **product-requirements.md** - プロダクト要求定義書
- **backlog.md** - プロダクトバックログ（機能単位の全体俯瞰。行と優先度は意図、状態列は `/add-feature` が機械更新）
- **functional-design.md** - 機能設計書
- **architecture.md** - アーキテクチャ設計書
- **repository-structure.md** - リポジトリ構造定義書
- **development-guidelines.md** - 開発ガイドライン
- **glossary.md** - ユビキタス言語定義

### 作業単位のドキュメント(`.steering/`)

特定の開発作業における「今回何をするか」を定義。作業ごとに新規作成し、履歴として保持する。

- `requirements.md`: 今回の作業の要求内容
- `design.md`: 変更内容の設計
- `tasklist.md`: タスクリスト
