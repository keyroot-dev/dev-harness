## スペック駆動開発の基本原則

### 基本フロー

1. **ドキュメント作成**: 永続ドキュメント(`docs/`)で「何を作るか」を定義
2. **作業計画**: ステアリングファイル(`.steering/`)で「今回何をするか」を計画
3. **実装**: tasklist.mdに従って実装し、進捗を随時更新
4. **検証**: テストと動作確認
5. **更新**: 必要に応じてドキュメント更新

### 重要なルール

#### 承認ゲートの方針（正本）

**「いつ人間の承認で止まり、いつ自走するか」はここで一元的に定義する。** 各コマンド・スキルはこの方針を再定義せず、これに従う。

- **セットアップ（`/setup-project`）**: 熟慮が必要なため、意図を固める点と、動く土台を確認する点に承認ゲートを置く。
  - **ゲート1（意図）**: 起点となる `product-requirements.md` は**作成後に必ずユーザー承認を待つ**。
  - PRD承認後、`architecture.md` の芯（依存の向き・レイヤ境界・技術スタック）と `docs/backlog.md`（機能単位の全体俯瞰）を生成する。どちらも意図（intent）なのでPRDを真実の源として連続生成してよい。
  - **ゲート2（動く土台）**: 続けて Walking Skeleton（動く縦切り1本＋CI・フィットネス関数が緑）を実装し、**動くことをユーザーに見せて一拍置く**。
  - **描写（description）ドキュメント**（`repository-structure.md`・`functional-design.md`・`glossary.md`・`development-guidelines.md`）は、骨格が動いた**後**に、コードを真実の源として導出／種まきする。憶測でコードの前に書かない（理由は「ドキュメント管理の原則（正本）」を参照）。
- **実装（`/add-feature`）**: 計画から完了まで**自走する（途中でユーザー判断を仰がない）**。
  - 判断に迷う事態が起きても、`steering` の「タスク完遂規律（正本）」に従って自律的に対処する。
  - 実装後、育つ描写ドキュメント（機能設計・用語集等）を、その機能ぶんだけ追記して育てる。

> 使い分けの理由: 意図（何を・なぜ）は方向性を決めるので人間が舵を握り、動く土台も一度確認する。描写（どこに何が）はコードから導出すべきもので、実装はその意図に従う「実行」なので規律に沿って一気通貫で進める。

#### 承認ゲートでの伝え方

承認ゲート（上記「承認ゲートの方針（正本）」で定義した2点）に達したら、必ず立ち止まり、明確に伝える。ゲート以外（アーキ芯や、骨格後の描写ドキュメント生成）では止まらず連続で進めてよい。

**ゲート1（意図＝PRD）で待つとき**:
```
「product-requirements.md の作成が完了しました。内容（意図）を確認してください。
承認いただけたら、アーキテクチャの芯と Walking Skeleton に進みます。」
```

**ゲート2（動く土台＝Walking Skeleton）で待つとき**:
```
「Walking Skeleton が動く状態になりました（CI・フィットネス関数も緑）。
実行して確認してください。承認いただけたら、骨格を元に残りの描写ドキュメントを生成します。」
```

#### 実装前の確認

新しい実装を始める前に、必ず以下を確認:

1. CLAUDE.mdを読む
2. 関連する永続ドキュメント(`docs/`)を読む
3. Grepで既存の類似実装を検索
4. 既存パターンを理解してから実装開始

#### ステアリングファイル管理

作業ごとに `.steering/[YYYYMMDD]-[タスク名]/` を作成:

- `requirements.md`: 今回の要求内容
- `design.md`: 実装アプローチ
- `tasklist.md`: 具体的なタスクリスト

命名規則: `20250115-add-user-profile` 形式

#### ステアリングファイルの管理

**作業計画・実装・検証時は`steering`スキルを使用してください。**

- **作業計画時**: `Skill('steering')`でモード1(ステアリングファイル作成)
- **実装時**: `Skill('steering')`でモード2(実装とtasklist.md更新管理)
- **検証時**: `Skill('steering')`でモード3(振り返り)

詳細な手順と更新管理のルールはsteeringスキル内に定義されています。

## ディレクトリ構造

### 永続的ドキュメント(`docs/`)

アプリケーション全体の「何を作るか」「どう作るか」を定義:

#### 下書き・アイデア（`docs/ideas/`）
- 壁打ち・ブレインストーミングの成果物
- 技術調査メモ
- 自由形式（構造化は最小限）
- `/setup-project`実行時に自動的に読み込まれる

#### 正式版ドキュメント
- **product-requirements.md** - プロダクト要求定義書
- **backlog.md** - プロダクトバックログ（機能単位の全体俯瞰。行と優先度は意図、状態列は `/add-feature` が機械更新）
- **functional-design.md** - 機能設計書
- **architecture.md** - アーキテクチャ設計書
- **repository-structure.md** - リポジトリ構造定義書
- **development-guidelines.md** - 開発ガイドライン
- **glossary.md** - ユビキタス言語定義

### 作業単位のドキュメント(`.steering/`)

特定の開発作業における「今回何をするか」を定義:

- `requirements.md`: 今回の作業の要求内容
- `design.md`: 変更内容の設計
- `tasklist.md`: タスクリスト

## 開発プロセス

### 初回セットアップ

1. このテンプレートを使用
2. `/setup-project` で PRD（意図）を作成・承認 → アーキ芯 → Walking Skeleton（動く土台）→ 骨格を元に描写ドキュメントを導出（承認ゲートの方針・ドキュメント管理の原則に従う）
3. `/add-feature [機能]` で機能実装

### 日常的な使い方

**基本は普通に会話で依頼してください:**

```bash
# ドキュメントの編集
> PRDに新機能を追加してください
> architecture.mdのパフォーマンス要件を見直して
> glossary.mdに新しいドメイン用語を追加

# 機能追加(定型フローはコマンド)
> /add-feature ユーザープロフィール編集

# 進捗の確認・PM向け報告(全体俯瞰は docs/backlog.md を開くだけでも可)
> /progress-report
> 進捗どう?

# 詳細レビュー(詳細なレポートが必要なとき)
> /review-docs docs/product-requirements.md
```

**ポイント**: スペック駆動開発の詳細を意識する必要はありません。Claude Codeが適切なスキルを判断してロードします。

## ドキュメント管理の原則

### 意図と描写の分離（正本 / Single Source of Truth）

**「どのドキュメントをコードの前に書き、どれをコードの後に書くか」はここで一元的に定義する。** 各スキル・コマンドはこの原則を再定義せず、これに従う。

ドキュメントには2種類ある。**この2種類を、書くタイミングで扱い分ける。**

- **意図（intent）ドキュメント = コードの「前」に書く**
  - 該当: `product-requirements.md`、`architecture.md` の芯（依存の向き・レイヤ境界・技術スタック）
  - 内容: 何を・なぜ作るか、制約、受け入れ条件、インターフェース、やらないこと
  - 理由: 意図を前もって詳しく与えると、LLMの生成品質が上がる（スペック駆動 / コンテキストエンジニアリング）。ここはケチらず詳しく書く。

- **描写（description）ドキュメント = コードの「後」に、コードから導出して書く**
  - 該当: `repository-structure.md`、`functional-design.md`、`glossary.md`、`development-guidelines.md`（骨格の設定から種を採る部分）
  - 内容: どこに何があるか、現在のモジュール構成、実際に使われているドメイン用語
  - 理由: 存在しないコードの描写を先に書くと「詳細な嘘」になり、それを真実として読んだLLMが確信を持って間違える（context poisoning）。**描写は真実でなければ武器にならない。** 真実の源はコードであり、描写はそこから導出して鮮度を保つ。

- **ハイブリッド: バックログ（`docs/backlog.md`）**
  - 行の存在と優先度は意図（PRD承認後に導出・人が舵を握る）。**状態列だけは描写**であり、`/add-feature` が着手・完了時に機械更新する（手で `[x]` にしない）
  - 進捗報告はここと `.steering/*/tasklist.md` から**都度導出**する（`/progress-report`）。報告書ファイルは保存しない — `docs/` は頻繁に更新されない北極星のみを置く場所であり、時点スナップショットは git 履歴からいつでも再導出できる

> 一言原則: **意図は分厚く前に、描写は真実として後から。** 真実を伴わない詳細は、規模を伴う毒（context poisoning）になる。

**描写の鮮度は機械で守る**: 描写ドキュメントと実コードの乖離（ドリフト）は、放置すればLLMへの毒になる。`scripts/check-config.sh` が参照リンク切れを機械検出するのと同じ思想で、描写と実装の乖離はフィットネス関数／CIで検出し、腐敗を防ぐ。進捗の乖離（backlog の状態列と tasklist の実態のズレ）は `scripts/progress.sh` が検出する。

### 永続的ドキュメント(`docs/`)

- 基本設計を記述
- 頻繁に更新されない
- プロジェクト全体の「北極星」

### 作業単位のドキュメント(`.steering/`)

- 特定の作業に特化
- 作業ごとに新規作成
- 履歴として保持
